= Server container log shipping

[abstract]
How to ship logs out of the Couchbase Server container.

This documentation is primarily focused on log shipping from the Couchbase Server image, it is not related to the actual configuration of the logging within those images. 
For details on Couchbase Server logging refer to that xref:server:manage:manage-logging.adoc[documentation].

.Basic overview of logging
image::logging-sidecar-overview.png[]

As you can see in the documentation, the Couchbase Server processes log to various files. 
This is all within the single Couchbase Server container and created in the (mutually exclusive) `default` or `logs` volume.

A Fluent Bit sidecar can be enabled to read the files in this volume within a pod and ship the logs out to follow standard Kubernetes practices. 
This then enables integration with other log monitoring or collection technologies deployed within a Kubernetes cluster.

[IMPORTANT]
====
A persistent volume is required to support log shipping. 
Otherwise the logs are all held within the server container so are inaccessible to the sidecar.
====

The logging sidecar uses the official Fluent Bit image by default. 
The operator will default the configuration of this sidecar so that it ships the audit log to standard output, however this configuration can be modified via a simple `Secret`. 
Anything supported by Fluent Bit is available to https://docs.fluentbit.io/manual/administration/configuring-fluent-bit[configure^].
A `Secret` is used in case any credentials or similar sensitive information is required for the log shipper.

Note that this approach also supports complete substitution of the log shipping technology and configuration if so required.
The sidecar could use an entirely different implementation configured appropriately.

By default the Couchbase Autonomous Operator will provide a log configuration that ships the audit log from the Couchbase Server image to the standard console output of the `logging` sidecar.
A user can access this via the traditional `kubectl -n <namespace> logs <pod> logging` command or hook it into any third party tooling as usual.

== Auditing
Logging encompasses audit logging and a more general audit configuration via the standard Couchbase Server REST API.
The CRD provides the various values that are passed through to the API.

One other issue with audit logging is that the rotated log files are never removed currently.
It is the responsibility of the administrator to clean these up. 
With the recent update to ship audit logs by default, it is also now possible to run a sidecar to cleanup the rotated logs as well. 
This can only be done if the logging sidecar is also enabled otherwise logs could be lost. 
There is no need to maintain the rotated log files as these will have been already shipped via the logging sidecar.

== Deployment

Auditing and logging can be enabled using the standard `CouchbaseCluster` CRD. 
The fields and defaults are shown below:

[source,yaml]
----
apiVersion: couchbase.com/v2
kind: CouchbaseCluster
spec:
  logging:
    server:
      enabled: false
      manageConfiguration: true
      configurationName: “fluent-bit-config”
      sidecar:
        image: “fluent/fluent-bit:1.7”
        configurationMountPath: “/fluent-bit/etc/”
    audit:
      enabled: false
      disabledEvents: []
      disabledUsers: []
      rotation:
        interval: “15m”
        size: “20Mi”
      garbageCollection:
        sidecar:
          enabled: false
          image: "busybox:1.32.1"
          age: “1h”
          interval: “20m”
----

The only required fields are the `enabled` ones set to true to turn the auditing and logging on, all fields are defaulted as indicated.

Customization of the images used is available, e.g. to add certificates for self-signed endpoints internally.

The `Secret` used to configure the logging sidecar is specified by the `configurationName` parameter along with its mount path into the container.

There is deliberately minimal support for configuration of the audit cleanup sidecar. 
Only the image used can be configured although this still opens up some security concerns.

== Logging configuration

As indicated above, the Couchbase Autonomous Operator will default logging configuration if it is not provided. 
All configuration is provided via the `configurationName` `Secret` in the namespace.

This `Secret` is used to https://kubernetes.io/docs/concepts/configuration/secret/#restrictions[populate the volume mounted into the sidecar^] and then picked up by Fluent Bit. 
As such it supports any and all configuration files required.

The xref:tutorial-couchbase-logging.adoc[tutorial] shows an example of populating it with a custom configuration. 

The key point is that the `Secret` is only picked up when a pod is started so a custom one must be available at this point to be used. 
Changing the `Secret` will update the volume in the sidecar but unfortunately Fluent Bit does not currently https://github.com/fluent/fluent-bit/issues/365[support dynamic reload^] of its configuration.
There are some options to support dynamic configuration, which may be needed for things like certificates or credential refresh:
1. Use a https://github.com/kubesphere/fluent-bit[custom image based on Fluent Bit^] that includes an `inotifywait` or similar process to watch for configuration changes and restart Fluent Bit.
2. Use a downstream aggregator that can support dynamic configuration (e.g. Fluentd).
image::logging-dynamic-config.png[]
3. Manually restart on configuration change.
4. Use another technology for the sidecar that does support dynamic configuration.
5. Use instance roles or similar to allow a static configuration with the authentication managed by the infrastructure.

[IMPORTANT]
====
To use a custom configuration, make sure to set `manageConfiguration: false` in the CRD.
If set true (or omitted so it will default to true) this means the Couchbase Autonomous Operator will reconcile its own configuration by overwriting any custom configuration present.

The `Secret` used for configuration must be present for the `Pod` to start so be aware of that.
====

== Example configuration
An example configuration showing the CRD with events and users provided to the optional audit fields:

[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: cb-example-auth
type: Opaque
data:
  username: QWRtaW5pc3RyYXRvcg== # Administrator
  password: cGFzc3dvcmQ=         # password
---
apiVersion: couchbase.com/v2
kind: CouchbaseEphemeralBucket
metadata:
  name: default
---
apiVersion: couchbase.com/v2
kind: CouchbaseCluster
metadata:
  name: cb-example
spec:
  logging:
    server:
      enabled: true
    audit:
      enabled: true
      disabledEvents: 
        - 8243
      disabledUsers: 
        - "@eventing/local"
        - "@cbq-engine/local"
      garbageCollection:
        sidecar:
          enabled: true
  image: couchbase/server:6.6.0
  security:
    adminSecret: cb-example-auth
  buckets:
    managed: true
  servers:
  - size: 3
    name: all_services
    services:
    - data
    - index
    - query
    - search
    - eventing
    - analytics
    volumeMounts:
      default: couchbase 
  volumeClaimTemplates: 
  - metadata:
      name: couchbase 
    spec:
      storageClassName: standard 
      resources: 
        requests:
          storage: 1Gi
----
